# Spreadsheet Sync and Calculation Boundaries

> 本文定义 OpenExcel 的表格状态、增量同步和 Excel 语义计算边界。
> 它是长期约束文档，不记录某次改动的过程，也不替代具体模块的实现文档。

## 1. 核心结论

OpenExcel 必须把完整快照和增量变更分开处理：

```text
完整 Sheet 快照：首次加载、导入、导出、恢复、同步失败时的兜底
Sheet Mutation：可明确表达的单元格、格式和结构变更的实时传输
```

增量同步的目标是减少网络传输和前端刷新范围，**不是把 Excel 的计算引擎搬到服务端**。

服务端可以读取快照、应用一个明确的 mutation、保存结果，但不得因为存储改造而重新实现
公式重算、筛选、排序、填充、引用传播或其他 Excel 原生语义。

## 2. 状态归属

| 状态或行为 | 权威组件 | 服务端职责 |
| --- | --- | --- |
| 单元格输入和编辑交互 | FortuneSheet / Web | 保存编辑结果 |
| 公式重算和依赖传播 | FortuneSheet / Web | 保存公式和前端产生的缓存结果，不负责通用重算 |
| 筛选、排序、隐藏行 | FortuneSheet / Web | 持久化组件产生的状态 |
| 单元格格式和合并 | FortuneSheet / Web；AI 可发出明确 mutation | 合并或保存结构化变更，不推导 Excel 语义 |
| 行列尺寸、冻结、缩放等视图状态 | FortuneSheet / Web | 持久化对应 Sheet 配置 |
| XLSX/CSV 导入导出 | `packages/core` 和格式适配器 | 调用适配器并保存结果 |
| AI 写入工具 | `packages/server` 的工具适配器 | 应用明确的内容 mutation；不做通用计算 |
| 持久化和权限 | `packages/server` | 授权、版本控制、快照读写和冲突处理 |

这条边界意味着：服务端的 `apply mutation` 是数据合并操作，不是 Excel 计算操作。

## 3. 当前数据模型

当前 `Sheet` 的内容和配置以 JSON 字符串快照存储：

- `uploadedData` 保存真实存在的 FortuneSheet 单元格；
- `config` 保存 Sheet 级配置；
- `columns` 保存列布局等元数据；
- 图表等独立对象由各自的持久化模型管理。

`uploadedData` 是一个快照字段，而不是单元格级数据库。除非有明确的性能证据，不能为了追求
理论上的数据库局部写入而先引入逐单元格表、后端公式引擎或第二套 Excel 对象模型。

保留快照并不妨碍增量传输。服务端可以按以下方式工作：

1. 读取当前 Sheet 快照；
2. 应用已授权且版本匹配的 mutation；
3. 保存更新后的快照；
4. 返回 mutation 和新版本号。

这种方式会在数据库内部重写快照，但网络请求仍然可以只携带变化部分。

## 4. Mutation 规则

Mutation 是跨 AI、Web 和服务端的变更协议。外部坐标使用 Excel 视觉坐标；进入存储层后统一
转换为内部坐标。每个 mutation 应包含目标 Sheet、操作内容、基于的版本和幂等标识。

实际写入协议统一为：

```ts
type SheetCommand =
  | { kind: "mutation"; mutationId: string; sheetId: number; baseRevision: number; mutation: SheetMutation }
  | { kind: "replaceSnapshot"; mutationId: string; sheetId: number; baseRevision: number; snapshot: SheetSnapshot };
```

这不是两个保存 API，而是同一个命令入口的两个 payload。两种命令都经过相同的权限校验、版本
条件更新、命令指纹校验、幂等 receipt 和 revision 返回流程。

建议的操作边界如下：

```text
writeCells       单元格值和公式
clearCells       清除单元格内容，保留未涉及的格式属性
formatCells      单元格格式的局部 patch
merge/unmerge    合并状态和对应单元格标记
sheetConfig      冻结、筛选、缩放、默认尺寸等 Sheet 级配置
structure        行列或 Sheet 结构变更
```

Mutation 只描述调用方已经确定的变化。服务端不得根据一个值变化自行推断筛选结果、公式依赖
或其他未包含在 mutation 中的 Excel 行为。

### 4.1 单元格内容

内容 mutation 可以携带值、公式以及已知的显示缓存。修改单元格内容时，不得覆盖该单元格
未参与本次操作的格式、批注或其他属性。

清除内容也不等于删除整个单元格对象：如果单元格还包含格式或合并标记，应保留这些非内容属性。

### 4.2 单元格格式

格式属于单元格数据的一部分，但必须使用局部属性合并：

```json
{
  "type": "format",
  "cells": [
    {
      "row": 1,
      "col": 2,
      "patch": { "bg": "#FFFF00", "fc": "#000000", "bl": 1 }
    }
  ]
}
```

没有出现在 `patch` 中的属性保持不变；使用明确的 `null` 或 `clearFormat` 语义清除属性。
范围格式可以使用范围操作，但服务端只应展开和合并明确指定的格式字段。

### 4.3 Sheet 级配置

筛选、冻结、缩放、默认行高、默认列宽等状态不应伪装成单元格 mutation。它们属于 Sheet
配置，应使用配置 patch 或独立的结构化操作。

当前 FortuneSheet 负责筛选条件、隐藏行以及 `filter`、`filter_select` 的行为；OpenExcel
负责保存和恢复这些状态，并在 XLSX 导入导出时做格式转换。OpenExcel 不应再实现独立筛选器。

## 5. 公式边界

公式必须区分“表达式”和“缓存结果”：

- 公式表达式属于持久化数据；
- 显示值或计算结果是缓存；
- 浏览器编辑器负责交互时的公式重算和依赖传播；
- 服务端保存浏览器提供的受影响单元格结果，但不负责通用公式重算。

手动编辑公式后，如果 FortuneSheet 重新计算了多个依赖单元格，客户端必须把这些受影响的
缓存变化一并保存。不能只保存公式单元格，然后要求服务端推导依赖结果。

AI 目前只能在已知结果时写入缓存值。AI 工具不应为了生成完整缓存而实现一套独立的 Excel
计算器。若未来需要 AI 修改公式并立即得到可靠的依赖结果，应让现有编辑器或专门的计算运行时
产生结果，再把结果回写为快照或 mutation；这属于明确的新计算能力，不能隐式塞进存储层。

## 6. 哪些操作可以增量，哪些必须兜底

### 6.1 优先使用 mutation

以下操作可以使用 mutation，前提是编辑器或调用方能明确提供受影响范围：

- AI 写入或清空单元格；
- 简单的手动单元格值修改；
- 明确的单元格格式修改；
- 明确的合并和取消合并；
- 明确的 Sheet 配置字段修改。

### 6.2 保留完整快照兜底

以下操作不得仅凭猜测生成局部 mutation：

- 排序导致大量单元格重排；
- 筛选条件和隐藏行变化无法从事件中可靠提取时；
- 行列插入、删除或移动；
- 公式导致大量依赖单元格重新计算；
- 表格、透视表、图表、数据验证等复杂对象变化；
- 编辑器只提供最终状态，没有提供可信的操作信息；
- mutation 应用失败、版本冲突或客户端状态不完整。

这些情况可以继续提交当前 Sheet 的完整稀疏快照。完整快照不是失败路径，而是保证 Excel
语义正确性的正式同步方式。

如果计算出的 mutation 大小接近或超过完整快照，或者无法确认所有受影响状态都已包含，应该
自动选择完整快照，而不是强行压缩成可能丢状态的 delta。

## 7. AI 与 Web 的同步

AI 工具的返回结果应包含：

- 操作摘要；
- 受影响的 Sheet 标识；
- 可应用到现有前端状态的 mutation；
- 必要时的局部预览。

前端收到 mutation 后，应优先局部更新当前工作簿。只有 mutation 无法应用、Sheet 版本不匹配
或本地状态缺失时，才重新获取完整工作簿。

AI 工具的数据库写入和前端同步是两个不同问题：服务端可以为了当前快照模型而重写完整
`uploadedData`，但聊天流不应因此携带完整 Sheet。

手动编辑可以逐步引入 mutation，但不能因为 AI 已有 mutation 协议，就把 FortuneSheet 的所有
编辑事件都假设成简单单元格修改。对于没有可靠操作元数据的事件，继续使用完整快照。

## 8. 版本、并发和失败处理

每个可持久化 mutation 都应基于 Sheet 版本应用：

- 客户端发送 `baseRevision`；
- 服务端在同一事务中校验并递增版本；
- 重复的 mutation 使用幂等标识处理；
- 请求成功但响应丢失后，使用相同 `mutationId` 重试必须返回第一次执行结果；
- 版本不匹配时不得静默覆盖其他用户的修改；
- 客户端收到冲突或 patch 失败时重新获取完整 Sheet；
- 快照保存和 undo 快照必须继续使用同一份 canonical cell model。

`updatedAt` 只能表示更新时间，不能替代 mutation 的版本号或冲突检测。

幂等命令指纹只覆盖 `mutationId`、`sheetId`、命令类型和 mutation/snapshot payload，不覆盖
`baseRevision`。版本号是首次执行时的并发控制前置条件；同一 `mutationId` 携带相同操作但使用已推进的
新版本重试时，服务端仍应返回原始 receipt。若同一 `mutationId` 携带不同操作 payload，则必须返回幂等冲突。

## 9. 当前实现状态与已知风险

本节记录统一 Sheet 命令链路落地后的审查结论。它不是新的协议，也不允许形成另一套
独立的保存系统；后续实现必须继续收敛到本文件定义的 `SheetCommand` 入口。

### 9.1 当前已经完成的部分

- `packages/core` 已定义 Sheet command、mutation、snapshot 以及纯函数 mutation applier。
- Server 已提供统一的 `executeSheetCommand` 应用入口，负责权限、版本、事务和幂等 receipt。
- AI 的写入工具已经通过 mutation command 进入统一入口。
- Web 已经接入统一命令入口，但当前手动编辑仍把 `onChange` 的最终状态提交为
  `replaceSnapshot`。

因此目前的传输收益是局部的：AI 的明确单元格操作可以只传 mutation；Web 手动编辑尚未
真正接入 mutation，单个单元格编辑仍可能上传整张 Sheet 快照。只要 FortuneSheet 还没有
可靠的操作级事件，继续使用 `replaceSnapshot` 是正确的安全兜底，不应通过前后快照猜测
用户操作。

### 9.2 高风险问题

#### A. Web 手动编辑尚未实现增量 mutation（P1）

当前 Web 的 `handleChange` 只能可靠拿到最终工作表状态，因此统一链路虽然已经接通，
但手动编辑没有获得增量传输收益。大表中一次单元格编辑会产生较大的请求体，并增加延迟、
请求超限和保存队列积压的概率。

这不是 `revision`、幂等或“增量保存”架构本身的问题，而是编辑器没有提供可信操作事件。
下一步应在 FortuneSheet/fork 中增加正式的值、公式和清除事件适配；在事件不足时，复杂
操作和不确定状态继续使用同一个 `replaceSnapshot` 命令作为永久兜底。

#### B. 合并会丢失非左上角单元格的格式（P1）

`applyMerge` 当前会把合并区域中非左上角单元格替换成只含合并标记的对象。这样会丢失
背景色、字体、边框等非内容属性，违反“合并不修改未参与变更的样式”的 mutation 契约。

该问题不是增量同步机制的必然代价。它来自合并 mutation 的具体实现，且统一到 Core 后
应只保留一份修复后的语义：非左上角单元格应清除内容/公式，但保留其余格式和合并所需
的标记。必须补充带格式区域的合并回归测试。

#### C. 合并状态存在两个可能不一致的来源（P1）

当前 canonical 快照包含 FortuneSheet 的合并信息，但 Web 读取时仍可能在快照中没有合并
信息时回退到 `Sheet.merges`。统一命令提交会更新快照和 revision，却没有同步更新这个旧
字段，因此“合并 → 取消合并 → 刷新页面”可能重新显示旧合并。

这属于存储权威未收口的问题，不是 mutation 传输模型的问题。长期唯一权威应是 canonical
Sheet snapshot；应移除旧 `Sheet.merges` 的读取回退，并在迁移期间明确处理历史数据。若暂时
保留该字段，任何合并命令都必须在同一事务中同步维护它，且必须有刷新后的集成测试。不得
让新代码继续扩大两个来源的兼容逻辑。

#### D. Undo 仍有绕过统一 Sheet 命令入口的写入（P2）

Undo 当前直接更新 Sheet 快照并递增 revision，没有复用 `executeSheetCommand` 的统一版本、
receipt 和结果语义。这不会影响普通 AI/Web 命令的网络传输，但会使 undo 成为另一条 Sheet
写入语义，长期可能与并发、审计和前端局部投影不一致。

Undo 应作为统一命令应用服务之外的专用编排层：它可以构造 `replaceSnapshot` command，
再调用同一个 Sheet 执行入口；工作簿/Sheet 的创建、删除等结构操作仍可保留各自的应用服务。

### 9.3 问题归因

| 问题 | 是否由增量机制天然导致 | 当前归因 |
| --- | --- | --- |
| Web 编辑仍传完整快照 | 否 | 编辑器缺少可靠操作级事件，当前选择了安全兜底 |
| 合并丢失格式 | 否 | Core 合并实现不完整，属于可修复逻辑错误 |
| 刷新后旧合并复现 | 否 | canonical snapshot 与 `Sheet.merges` 双来源未收口 |
| Undo 绕过统一入口 | 否 | 统一写入迁移尚未完成 |
| AI mutation 减少请求体 | 是，已生效 | AI 已使用统一 mutation command |

结论：增量 mutation 的方向和统一命令架构仍然合适，不需要推倒重来。当前风险主要是
具体 mutation 语义、存储权威和剩余写入路径没有完全收口。继续增加第二套“增量保存 API”
会重新制造双轨问题，应优先修复 Core 和统一应用入口。

## 10. 后续实施顺序

按用户体验和数据正确性排序：

1. 修复 `applyMerge`，保留非左上角单元格的格式，并增加带格式合并测试。
2. 确定合并信息的唯一权威为 canonical snapshot；处理历史 `Sheet.merges` 数据后移除
   读取回退，或在明确的短期迁移窗口内保证事务级同步。
3. 增加合并、取消合并、刷新页面的真实链路集成测试，并覆盖 receipt 重放和 revision 冲突。
4. 将 Undo 改为构造统一的 `replaceSnapshot` command，移除直接更新 Sheet 的写入路径。
5. 在 FortuneSheet/fork 中提供可靠的单元格值、公式和清除事件。
6. 仅将这些可确认事件接入 `mutation`；排序、筛选、插入删除、复杂公式依赖和状态不完整
   的场景继续使用同一入口的 `replaceSnapshot`。
7. 通过真实大表数据统计请求字节、保存延迟和失败率，再决定是否需要快照压缩或其他存储
   优化；在证据出现前不引入逐单元格数据库。

## 11. 变更前检查清单

任何涉及 Sheet 存储、编辑事件、AI 工具或同步协议的改动，都必须先回答：

1. 这项行为的语义权威是 FortuneSheet、core、server 还是 agent？
2. 服务端是否只是应用已确定的状态，还是开始推导 Excel 语义？
3. 公式的表达式和所有受影响缓存结果是否都被保存？
4. 筛选、排序、隐藏行、冻结和 Sheet 配置是否仍由 FortuneSheet 管理？
5. 格式、合并、列宽等非内容状态是否包含在同步范围内？
6. 编辑器是否真的提供了可靠的 mutation 信息？如果没有，是否保留完整快照？
7. mutation 失败、版本冲突和本地状态不完整时，是否能回退到完整 Sheet？
8. 导入、重新加载、XLSX 导出和 undo 是否仍基于同一份快照模型？

## 12. 不变量

- 不为了减少数据库写入而在服务端复制 Excel 计算引擎。
- 不为筛选、排序或公式依赖建立第二套隐式语义实现。
- 不让局部 mutation 覆盖未参与变更的单元格格式或对象状态。
- 不把完整快照从同步协议中删除；它是初始化、恢复和失败兜底的一部分。
- 不让 AI 工具直接决定前端状态结构；AI 只能产生受 schema 约束的 mutation。
- 不让手动编辑、AI 编辑和导入导出使用互不兼容的单元格模型。
- 不在没有版本控制和幂等处理的情况下启用跨请求 mutation。
